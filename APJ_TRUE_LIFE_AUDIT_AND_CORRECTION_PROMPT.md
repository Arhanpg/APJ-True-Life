### 2.6 🟡 RLS security guard doesn't actually check *which* Firebase project

Your README states: *"tokens from any other Firebase project or unknown issuer are
rejected at the DB level."* I pulled the live function definition to verify:
```sql
SELECT (
  (auth.jwt()->>'iss') = 'https://bwozwxrzotnlajutxupm.supabase.co/auth/v1'
  OR (
    (auth.jwt()->>'iss') LIKE 'https://securetoken.google.com/%'
    AND (auth.jwt()->>'aud') IS NOT NULL

```
This accepts a valid ID token from **any** Firebase project on earth — the `LIKE
'.../%'` wildcard doesn't pin a specific project, and `aud IS NOT NULL` is true for
every legitimate Firebase token regardless of project. In practice you're likely
also relying on Supabase's Third-Party Auth dashboard config (Authentication →
Third-Party Auth) to only accept your specific project — but that's a second,
separate control, and the DB-level function should not silently be weaker than
documented. I didn't fix this myself because it needs your real Firebase Project
ID, which I don't have — see §3 and §4.
